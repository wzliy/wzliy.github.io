Redis内核探究

主从复制

为了实现Redis的高可用，我们需要保证数据尽量少丢失，服务尽量少中断，AOF 和 RDB 保证了前者，而对于后者，Redis 的做法就是增加副本冗余量，将一份数据同时保存在多个实例上，也就是通过集群的方式搭建Redis服务。

那如何保证多实例间数据的一致性？

Redis提供了主从库模式，以保证数据副本的一致性，主从库之间采用的是读写分离的方式。

主库负责写，从库负责读，主库将写入的数据分发到从库。

Redis实现分布式锁

Redis本身可以被多个客户端共享访问，正好就是一个共享存储系统，可以用来保存分布式锁。

分布式锁与单机锁既有相似性，又有分布式场景下的特殊要求

实现分布式锁的两个要求：

  • 分布式锁的加锁和释放锁的过程涉及多个操作，所以，在实现分布式锁时，我们需要保证这些锁操作的原子性。
  • 共享存储系统保存了锁变量，所以，我们需要考虑保证共享存储系统的可靠性，进而保证锁的可靠性。

分布式锁需要解决的问题

  • 互斥性，任何时刻，只能有一个客户端获取锁
  • 安全性，锁只能被持有该锁的客户端删除，不能被其他客户端删除
  • 死锁，要有机制避免某个客户端持有锁之后不可用，导致其他客户端一直获取不到锁
  • 容错，Redis某个节点宕机后，仍然能使用锁

基于单个Redis节点实现分布式锁

Redis可以用哪些单命令操作实现加锁操作：

加锁 SETNX

SETNX key value:如果key不存在，则创建并赋值，如果key已经存在，就不做任何设置

释放锁 DEL

删除锁变量

// 加锁伪代码

// 加锁
SETNX lock_key 1
// 执行业务逻辑
DO ThINGS
// 释放锁
DEL lock_key

不过，这种方式存在两个风险：

  1. 假如某个客户端在执行SETNX命令加锁之后，紧接着发生了异常，结果一直没有执行DEL命令释放锁，导致其他客户端一直拿不到锁
  2. 有可能当前客户端执行时，其他客户端释放了锁，导致当前客户端的锁失效，引起并发问题

第一个问题可以通过设置过期时间，第二个问题可以通过每个客户端给锁变量设置一个唯一值，用来标识当前客户端，在释放锁操作时，客户端判断当前锁变量的值是否和自己的唯一标识相等，只有相等的情况下才能释放锁。

所以我们可以通过SET命令来设置过期时间以及唯一值

通过给SET命令设置NX，可以达到和SETNX相同的效果，我们可以通过这个命令来实现加锁操作

// 加锁，unique_value 作为客户端唯一标识，PX 1000 设置过期时间
set lock_key unique_value NX PX 1000
